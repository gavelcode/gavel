# Rust

Gavel analyzes Rust code with [Clippy](https://doc.rust-lang.org/clippy/),
the official Rust linter. Unlike other languages where gavel runs a custom
wrapper, Rust delegates to the official
[`rust_clippy_aspect`](https://bazelbuild.github.io/rules_rust/rust_clippy.html)
from `rules_rust` — the same aspect used by the broader Bazel+Rust ecosystem.

| | |
|---|---|
| **Lint tool** | Clippy (from Rust toolchain via `rules_rust`) |
| **Upstream aspect** | `@rules_rust//rust:defs.bzl%rust_clippy_aspect` |
| **Gavel aspect** | `rust_clippy_submission_aspect` (converter wrapper) |
| **Archtest** | `rust_archtest_submission_aspect` |
| **Bazel rule kinds** | `rust_library`, `rust_binary`, `rust_test` |
| **SARIF suffix** | `.clippy.sarif` |
| **Converter** | `lint/lang/rust/clippy/converter/` |
| **Coverage** | LCOV via `bazel coverage` |

## Architecture: delegated aspect

Gavel's Rust linting uses a two-phase aspect pipeline:

1. **`rust_clippy_aspect`** (from `rules_rust`) runs first. It invokes
   `rustc` directly (not `cargo clippy`) with all inputs properly declared
   via `collect_inputs`. This is fully hermetic — no `no-sandbox`, no
   undeclared file reads. With `clippy_output_diagnostics=True`, it
   produces a `.clippy.diagnostics` file containing rustc JSON diagnostics.

2. **`rust_clippy_submission_aspect`** (gavel's) runs second via
   `requires = [rust_clippy_aspect]`. It reads the diagnostics JSON from
   `OutputGroupInfo.clippy_output`, converts it to SARIF using a Go
   converter binary, and emits the result via the `gavel_submissions`
   output group.

```
rules_rust clippy_aspect (hermetic, runs rustc)
    → .clippy.diagnostics (rustc JSON)

gavel rust_clippy_submission_aspect (converter only)
    → reads diagnostics JSON
    → writes .clippy.sarif (SARIF 2.1.0)
    → output group: gavel_submissions
```

This approach gives gavel the hermetic caching of `rules_rust` while
maintaining compatibility with gavel's SARIF-based pipeline.

### Why not a custom wrapper?

Other gavel languages (Go, Java, Python) use custom wrapper binaries that
invoke the linter as a subprocess. For Rust, this approach had a latent
bug: `cargo clippy` reads the entire crate directory (undeclared inputs),
so Bazel's action cache could serve stale results.

`rules_rust`'s aspect avoids this by using `rustc` directly with all
sources declared as inputs. Since `rules_rust` is already a dependency
(for the Rust toolchain), delegating to its aspect is both correct and
simpler than maintaining a custom wrapper.

## Caching behavior

| Change | Cache invalidation |
|--------|--------------------|
| Edit a `.rs` file | rules_rust re-runs clippy → new diagnostics → new SARIF |
| Upgrade Rust toolchain | All Clippy SARIFs re-generated (toolchain is an input) |
| Edit `clippy.toml` | rules_rust re-runs clippy (config is a declared input) |

Rust now has the cleanest caching story of all supported languages.
All inputs are properly declared by `rules_rust` — source files,
dependencies, toolchain, and config files. No `no-sandbox`, no
undeclared reads.

## Configuration

Clippy reads `clippy.toml` or `.clippy.toml` from the workspace root.
`rules_rust` declares the config file as an explicit aspect input —
changes to it correctly invalidate the cache (unlike other languages
that rely on the `gavel_lint_config` filegroup).

Additionally, the `gavel_lint_config` filegroup still captures these
files for consistency across the pipeline.

Required `.gavel/gavel.bazelrc` settings (generated by `gavel init`):

```
build:gavel-rust-clippy --@rules_rust//rust/settings:capture_clippy_output=True
build:gavel-rust-clippy --@rules_rust//rust/settings:clippy_output_diagnostics=True
```

## Known limitations

1. **No pinned version.** Clippy's version is tied to the Rust toolchain.
   Different workspaces may get different results depending on their
   Rust version.
2. **Diagnostics mode required.** The `clippy_output_diagnostics` setting
   must be enabled for gavel to receive structured JSON. Without it,
   the converter receives text output and produces zero findings.

## Converter reference

```
lint/lang/rust/clippy/converter/
├── converter.go       # Convert() function: rustc JSON → SARIF
├── converter_test.go  # Unit tests
├── BUILD.bazel        # Library target
└── cmd/
    ├── main.go        # CLI binary: --in <diagnostics> --out <sarif>
    └── BUILD.bazel    # Binary target used by the aspect
```

| Flag | Description |
|------|-------------|
| `--in` | Input file (rustc JSON diagnostics from rules_rust) |
| `--out` | Output file (SARIF 2.1.0) |
